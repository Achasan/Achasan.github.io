인터페이스와 다형성 (Interface)

2021.08.17

인터페이스 (Interface)

인터페이스는 추상클래스와 유사한 클래스이다. 인터페이스는 추상클래스처럼 추상메소드를 갖지만 추상메소드보다 추상화 정도가 높다. 또한 추상메소드가 일반 멤버변수, 생성자를 가질 수 있지만 인터페이스는 그렇지 않다. 오직 상수와 추상메소드만을 가질 수 있고, 다른 부분은 허용하지 않는다.

추상클래스를 미완성된 클래스라고 한다면, 인터페이스는 기본적인 설계도만을 가진 클래스라고 할 수 있다.

인터페이스를 작성하는 방법은 클래스와 같다. class 대신 interface를 작성한다는 점만 빼면.

interface 인터페이스이름 {
	public static final 타입 상수이름 = 값; // 필드에는 상수만 선언할 수 있다.
	public void 메소드이름(매개변수); // 추상메소드
}

인터페이스 멤버는 다음과 같은 변수가 있다.

💡 모든 멤버변수는 public static final 이어야하고, 이를 생략가능하다. (자동 인식) 모든 메소드는 public abstract 이어야하고, 이를 생략이 가능하다. (static 메소드, default 메소드 제외)

생략된 제어자는 컴파일 시에 컴파일러가 자동적으로 추가해준다.

인터페이스의 상속

인터페이스는 인터페이스로부터만 상속받을 수 있고, 클래스와는 다르게 다중상속이 가능하다.

여기서 주의해야할 점은 인터페이스끼리만 상속이 가능하다는 점이다. 인터페이스 간에는 여러 개의 상속을 받는 것이 가능하다. 자식 인터페이스는 부모 인터페이스의 정의된 멤버를 모두 상속받는다.

인터페이스의 구현

인터페이스도 추상클래스처럼 그 자체로는 인스턴스를 생성할 수 없고, 자신에 정의된 추상메소드를 구현하는 구현클래스를 작성해야한다. 인터페이스에 있는 추상메소드와 상수를 상속받기 위해서는 extends대신 implements를 작성하여 상속을 받는다고 생각하면 된다.

class 클래스이름 implenets 인터페이스이름 {}

만약 구현하는 인터페이스의 메소드 중 일부만 구현한다면 abstract를 붙여서 추상클래스로 선언하어야한다. (처음 듣는 내용)

abstract class Fighter implements Fightable {
	public void move(int x, int y) {  }
}

또한 클래스의 상속과 인터페이스의 구현을 동시에 할 수 있다.

class Fighter extends Unit implements Fightable { }

인터페이스를 이용한 다중상속

이전에 자바에서 클래스 다중상속을 지원하지 않는다고 했던 적이 있었다. 다중상속은 장점도 있지만 단점이 더 크다고 판단하여서 지원하지 않았는데, 다른 언어와 다르게 다중상속을 지원하지 않아 발생할 수 있는 단점들을 보완하기 위해 자바에서는 인터페이스를 다중상속할 수 있도록 하였다. (하지만 자바의 정석에서는 인터페이스 다중상속을 거의 사용하지 않는다고 언급한다.)

책에서는 가볍게 다루고 있으므로 다중상속에 대해서 자세히 들여보지는 않을 것 같다. 자바의 정석에서는 인터페이스가 단순히 클래스와 다르게 다중상속이 되는 수단으로 생각하는 것을 경계하고 있으며, 인터페이스를 사용하는 중요한 이유에 대한 개념을 중점적으로 다루고 있어서 넘어가도록 한다.

인터페이스의 장점

인터페이스를 사용하는 이유와 장점은 무엇이 있을까.

  1. 개발시간을 단축시킬 수 있다.
  2. 표준화가 가능하다.
  3. 서로 관계없는 클래스들에게 관계를 맺어줄 수 있다.
  4. 독립적인 프로그래밍이 가능하다.

인터페이스가 작성되면 이를 이용하여 프로그램을 작성하는 것이 가능한데, 메소드를 호출하는 쪽에서는 메소드의 내용에 고나계없이 선언부만 알 수 있기 때문이다. 인터페이스를 구현하는 클래스를 작성하는 동안 그 클래스가 작성될 때 까지 기다리지 않고 양쪽에서 동시에 개발진행이 가능하다는 점도 개발시간을 줄이는데 도움이 된다.

표준화가 가능하다는 점은, 프로젝트에 사용되는 기본 틀을 인터페이스로 작성하여 개발자들에게 인터페이스를 구현하여 프로그램을 작성하도록 함으로써 보다 일관되고 정형화된 프로그램이 가능하다는 점이 있다.

클래스 간에 상속관계에 있지 않고, 같은 조상클래스를 가지고 있지 않은 서로 아무런 관계도 없는 클래스들에게 하나의 인터페이스를 공통적으로 구현하도록 함으로써 관계를 맺어줄 수 있다.

인터페이스를 이용하면 클래스의 선언과 구현을 분리시킬 수 있어 독립적인 프로그램을 작성하는 것이 가능하다. 클래스와 클래스 간의 직접적인 관계를 인터페이스를 이용하면 간접적인 관계로 변경하면 영향을 미치지 않는 독립적인 프로그램 작성이 가능해진다.

지금은 인터페이스의 활용성에 대해서 의문이 있지만, 실무에서는 인터페이스를 통해 클래스를 표준화 시키고, 간접적인 관계를 맺어줌으로써 서로 영향을 받지 않고 독립적인 프로그램의 실행에 도움을 주는 클래스로 알고 있어야 할 것 같다. 나중에 학원에서 프로젝트를 진행하게 되면 알 수 있을 것 같기도 하다.

다형성(polymorphism)

다형성은 객체 지향 프로그램의 4대 특징 중 하나로, 여러가지 형태를 가질 수 잇는 능력을 의미한다. 자바에서는 한 타입의 참조변수로 여러 타입의 객체를 참조할 수 이도록 하여 다형성을 프로그램적으로 구현한다.

한마디로, 부모 클래스 타입의 참조변수로 자식 클래스의 인스턴스를 참조할 수 있도록 하였다는 뜻인데, 처음에는 이해하기가 어려워서 반복하여 살펴보는 것이 중요한 개념이다.

TV tv = new SmartTV();

SmartTV가 부모 클래스인 TV 클래스를 상속받았다고 했을 때, TV 타입으로 매개변수를 선언하고 SmartTV 객체를 생성할 수 있다.

지금까지는 인스턴스를 다루기 위해서 인스턴스의 타입과 일치하는 타입의 참조변수만을 사용했었는데, 인스턴스와 참조변수가 일치하지 않는 경우의 선언도 가능하며, 이 경우에는 참조변수와 인스턴스가 서로 상속관계에 있어야 가능하다.

그렇다면 굳이 참조변수와 인스턴스 객체를 다르게 생성하는 이유는 무엇일까? 위에서 정의한 다형성을 위해서이다. 아래 코드를 살펴보자.

Tv tv1 = new TV();
Tv tv2 = new SmartTV();

첫 줄의 코드는 참조변수와 인스턴스 객체가 일치하고, 두 번째 줄의 코드는 참조변수 Tv와 자식클래스인 SmartTV를 인스턴스 객체로 생성한 코드이다.

두 코드의 차이점은 무엇일까? 첫 번째 TV 객체는 TV 클래스 내에 있는 멤버와 메소드를 모두 사용이 가능하다. 두 번째 코드는 SmartTV가 TV클래스를 상속받았기 때문에 TV클래스의 멤버와 SmartTV의 멤버를 모두 사용가능할 것 같지만 아니다. SmartTV 객체를 생성하였지만 참조변수가 TV이기 때문에 SmartTV객체는 TV의 멤버만 사용이 가능하다. 참조변수의 타입에 따라서 사용할 수 있는 멤버의 개수가 달라지는 것이다.

여기서 유의해야될 점은 반대로 자식 클래스 참조타입에 부모클래스 객체를 생성하는 것은 불가능하다.부모클래스보다 자식클래스가 멤버가 더 많기 때문이다. 참조변수가 사용할 수 있는 멤버의 개수는 인스턴스의 멤버 개수보다 같거나 적어야한다.

참조변수의 형변환

참조변수도 형변환이 가능하다. 단, 서로 상속관계에 있는 클래스 사이에서만 가능하기 떄문에 자식 타입의 참조변수를 부모타입의 참조변수로, 부모타입의 참조변수를 자식타입으로의 형변환만 가능하다.

여기서 자식타입을 부모타입으로 형변환하는 것을 up-casting, 부모타입을 자식타입으로 형변환하는 것을 down-casting이라고 한다.

class Car {
	String color;
	int door;
	void drive() {
		System.out.println("Drive");
	}
	void stop() {
		System.out.prinltn("Stop");
	}
}

class FireEngine extends Car {
	void water() {
		System.out.println("Water");
	}
}

class Ambulance extends Car {
	void siren() {
		System.out.println("Siren");
	}
}

위의 코드에서는 Car라는 부모클래스를 만들고, 자식클래스인 FireEngine과 Ambulance를 생성하였다. 객체 지향 프로그래밍의 상속관계에는 형제관계가 없으므로 자식클래스들은 Car 클래스만 상속받고, 자식클래스끼리는 아무런 관계가 형성되어있지 않다.

여기서 형변환을 일으키는 코드를 작성해보았다.

Car car = null;
FireEngine fe = new FireEngine();
car = fe; // up-casting 발생

FireEngine fe2 = (FireEngine)car; // down-casting 발생 - 에러 발생

코드의 다섯 번째 줄인 FireEngine fe2 = (FireEngine)car; 에서 오류가 발생하는데, 이유는 FireEngine 객체를 참조하는 car 변수를 FireEngine으로 형변환하는 것은 좋았으나 문제는 참조하는 변수가 Car 타입인 car변수를 형변환하는 것은 불가능하기 때문이다.

위에서 자식참조타입에 부모객체를 대입하는 것은 불가능하다고 했었는데, Car 참조변수에 FireEngine으로 형변환을 시도하려고하니 에러가 발생한다. 이유는 부모클래스는 자식클래스보다 멤버가 적을 수 밖에 없는데, 부모가 가지지 않은 멤버를 사용하려고 하는 행위가 불가능하기 때문이다. 컴파일오류는 발생하지 않지만, 코드가 실행 될 때 ClassCastException이 발생한다. 캐스팅은 헷갈릴 수 있기 때문에 형변환이 가능한지 확인 후에 형변환을 해주어야한다. 이러한 형변환이 가능한지를 알려주는 instanceof연산자가 있으므로 이를 활용하는 것이 좋다.

instanceof 연산자

instanceof연산자는 참조변수가 참조하고 있는 인스턴스의 실제 타입을 알아보기위해 사용한다. instanceof의 왼쪽에는 참조변수를 넣어주고, 오른쪽에는 타입을 넣어준다. 연산의 결과는 boolean값인 true와 false이다. true일 경우 검사한 타입으로 형변환이 가능하다는 것을 뜻한다. 보통 조건문에 instanceof 연산자를 사용하여 true가 나오면 자동으로 형변환이 되도록 코드를 작성하는 편이다.
